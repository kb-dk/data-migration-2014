Migration of old applications to the Royal Library's content management environment

Jacob Larsen and Sigfrid Lundberg
The Royal Library
Søren Kierkegaards Plads
Copenhagen

Abstract -- This document aims att giving an overview of the various services provided by DUP, and what can be done in order to (i) secure this data for future use, (ii) incorporate their different user interfaces into the new Content Management System implemented by the Royal Library.
By the first point we understand that the data should be available in a portable format which should be able to withstand the wear and tear of time and outlive the technologies used for deploying them. By the second point we understand that the format used for storing the data should be such these data should be easily rendered in a user friendly manner from inside the CMS using industry standard technologies. It should also be easy to import the data into search engines such as Primo from ExLibris.

Introduction

User satisfaction is the prime goal of any Internet information service. Crucial to a positive user experience is the usability of the search and navigation systems combined with clear presentation of data. The user must obviously find the data that he or she percieves as relevant for the problems that triggerd his or her information seeking behaviour.

During the last five years or so, the call for ease of use of information systems has triggered new views on the use metadata. Earlier metadata systems were primarily used for supporting advanced search forms, whereas today the emphasis is to use metadata for the clustering of resources and for navigation.

The acquisition of the Primo search portal is an important step towards a more efficient use of the Royal Library's electronic resources, both those licensed and those produced in house. This paper describes a program that will

  • make digitized resources accessible from inside the new content managagement environment
  • make export of metadata from the Royal Library's digitization projects easier; data that may then be imported into Primo

Executive Summary of our Proposal

In this and the following section we propose actions. In order to simplify for readers we employ the typographic convention that, whenever we propose some thing the word propose appear in bold italics.

A majority of the digital resources provided by the Royal Library have lead a life independent from the main web site www.kb.dk having base.kb.dk as their home. Their age and quality differ widely; some of them are quite up to date and of good quality. Other are old and built using technologies no longer supported by the library's technical platform. Some of the oldest and poorest were built as interimistic solutions (i.e., they were never meant to be permanent). There are the five purely bibliographic databases: Dansk Juridisk Bibliografi, Dansk Sociologisk Bibliograpfi and Dansk Musik Fortegnelse. The services Håndskriftsafdelingens Erhvervelser and Håndskriftsafdelingens Brevdatabase belong to this category as well, but to us they seem a bit more important, since they actually catalogue some of our own collections. We propose that these two databases are converted, the former for inclusion in REX and the latter for producing a new seperate database comprising all letters, i.e., also those catalogued in REX. We propose that it should be searchable with respect to senders and recievers of messages. Furthermore, we propose that the remaining three are kept as they are for later inclusion in Primo at some later stage.

A second category of service is ones containing digitized material. These are H.C. Andersens Dagbøger, the two manuscript services HSK and MANUS and finally the various services providing sheet music, i.e., Lumbye Online, Hartman and MUS. It seems to us that they have been built without a proper understanding of the complexity of the content to be stored in them, and with insufficient guidance by functional specifications. The data models behind them has not been flexible enough to successfully accommodate the data which is there today, which have been shoe-horned into the base stretching the semantics of the database schemas far beyond their intended design. In particular neither of the manuscript databases have been capable of storing a proper table of content for the ditigized works, making it hard to navigate the resources in any other way than by browsing from image to image. It has further been impossible to assign metadata to parts of manuscripts. The databases contain HTML markup, among other things hard coded hypertext links back to various documents in the current web site, making it more or less impossible to integrate the data into a new refurbished web.

We propose the export these four services into a new format described below. That means that we will build new user interfaces for all of them (One application that can handle all four services). However, there will not be any new editorial interfaces. I.e., the Oracle implementations will still be used for data entry.

In our view, we need a capable and modern bibliographic database which need to be able to handle metadata of different level complexity, such as DCMI Terms, Metadata Object Description Schema (MODS) and MARCXML. We propose that Primo should be the main vehicle for searching this kind of metadata. A crude interimistic solution has to be provided until that software become available.

However, for the navigation and presentation of diverse kinds of digitized data, including text, sound and video, we need systems capable of handling metadata describing the interiors of these resources. This involves the search and navigation inside ordered (and sometimes overlapping) hierarchies of content objects. This requires that searching be combined with navigation inside complex resources. The data formats used should thus accommodate both traditional bibliographical data, but in addition to that it need to be able to describe the relations between sets of contents. The new data format should

  • be able to describe individual content objects as well as sets of such objects. Such descriptions should be able to include more or less arbitrary relations between objects.
  • be able to describe objects described in the same format, i.e., it should be able to use for collection level description.
  • be reusable for different purposes in house
  • be capable to live beyond the expected life span of the various KB current technical platform
  • make data reusable for external partners, e.g., via XML Web Services, OAI-PMH and similar protocols but also future emerging technologies.
  • be well documented.
  • form a basis for a sustainable hypertext linking policy (see below)

To our knowledge, the only format that fulfill these requirements is the Metadata Encoding and Transmission Standard (METS). In particular it is capable of describing more or less arbitrary relations between objects, and in addition assign metadata to each component. METS files may be transformeed into other formats for navigation of virtually any digital resource. Examples from other digital libraries include using it for generating SMIL for the synchronization of image and audio and to HTML for navigation text and images.

We propose that this is done for the purpose of cleaning the data and for generating a viewer that can operate inside the new web site with the technology available there. However we also propose that we should evaluate METS for a wider deployment in our digitization projects.

We propose that work level metadata are exported using OAI-PMH for use by Primo in house and by partners elsewhere.

Finally, we propose METS should be used for the description of collections of METS metadata, hence the requirement of a metadata system capable of collection level description.

How can these resources be integrated in the CMS

By integration we understand two things:

  1. Resources should be easy for the editors to use
  2. Resources should be easy for the user to find

Resources should be easy for editors to use

Scenario 1: An editor wants an index of certain collection of manuscripts in his webpage. He finds (not necessarily in the CMS) the METS file, describing the collection. Uploads the file into the CMS, in the folder where he wants the index. Chooses template ('Mets index'). Then he uploads the folder containing the METS files of the manuscripts. Chooses template ('Mets Navigator') on the folder. And now he has a webpage with manuscripts.

Scenario 2: Each METS file is uploaded once in a METS repository and the template property is set on this folder. This folder is used as a METS gallery in OpenCms. An editor can then choose which collection of manuscripts he wants, and include them into his page, using the gallery functionality.

The first solution gives the editor a little more work, but makes it more flexible for him when he wants to add a new METS file. The second option is easy to use for the editor, and gives an administrator the work of uploading METS files to the gallery, making sure that their content is described, and their names are meaningful.

Resources should be easy for the users to find

Every source is indexed with Lucene. An "agent" is made, to customize a subset search. In that way the editors can include a search form only searching in relevant material as they like, making it easy for our users to locate any resource.

What is needed in the CMS?

  • Two open cms templates
    1. One for displaying a single METS file
    2. One for displaying METS files describing METS files. (Might be the same as 1, with another css)
  • One search form agent (Could come from Primo)

A note on URI maintenance

URLs don't change, people change them, says Tim Berners Lee. There is no force of nature that makes URI invalid. If one wants cool URLs that don't change, then one need a good URI maintenance policy, like the one used by the World-wide Web consortium. The URI coolness was not an issue for the former colleagues of us who built the services, since coolness was not a part of the functional specifications for their work. We cannot preserve the URIs which is the fruit of their work, however we will try to invent a redirection scheme that will give useful hints to users. We propose that the Royal Library formulates persistence policy for the kb.dk domain, which should ensure that "perma-links" may be assigned to significant intellectual content published within our domain.

Digitization services

The following seven services are of very different qualities, age and other characteristics. For technical and other reasons we have to make a significant effort in order to migrate the first five of them, whereas this is less critical for the remaining two.

HSK (The previous generation manuscript application)

Contains 62 digitized, comprising about 11000 images.

Stake holders

  • DUP Developers - development and technical owners of the application
  • Manuscript Department

Current technology

Some objects are overloading the datamodel, which contains 4 tables of which only two are used.

Recommendations

The ideas behind this service have been deployed in quite a few services and applications (LUM, MUSMAN and MUS). The quality of all of them is wanting, and the databases contain a lot of dirty markup which makes the data hard to reuse without some investment in datamining and semi-automatic cleaning.

As regards the content of this database, it should ideally be merged with the content of MANUS, which, however, we see as a too large task to perform in connection with the proposed projects. Still we propose the building of a new joint application with the following properties.

  • Capability of exporting work level metadata for search and navigation within the new web site.
  • Have a flexible methods for describing the structure of the resources within the collection.
  • Be extensible enough to permit the transfer of other services to this framework.

We suggest that the METS should be used for this purpose. We would gain the following: First, a single application could deliver all four services, second we would get clean data that could be reused in our services or shared with other libraries, and finally we would be able to evaluate this metadata format for the possibility of future deployment in our cultural heritage services.

We have made a demonstrator that show that Mets will do the job for the data we possess. In spite of that there is risk that we find that METS will not be ideal for future deployment; if that will be the case the cost of is fairly small since the date is now clean and can be transformed into other formats or digested into a new database.

MANUS

The Royal Library's current manuscript services. It is a replacement of the HSK service.

  • Contains 645 digitized manuscripts
  • Altogether the service contains approx. 60000 images. Some 2300 of them have some form of transcription

Stake holder(s)

  • DUP Developers - development and technical owners of the application
  • Manuscript Department
  • Centre for Orientalia and Judaica
  • Music and Theatre Department

Current technology

  • Some transcriptions, notably the POMA manuscript are encoded using TEI. We do not know where the XML source is stored.
  • The service has a failry simple user interface but an advanced data model.
  • Known problems are that the service cannot handle works in multiple volumes and that texts in multiple languages cause problems (cf., Twiki page).

Recommendations

As noted above the HSK service should ideally be merged into this application, or ideally into the same framework as HSK. We hesitate to give any particular recommendations on this service, but give two possibilities (of which we propose the first):

  1. We do the translation as for HSK, and for POMA (and if there are any other TEI fulltexts) we build an interimistic structure somewhat like the one we built in the Gustaf Munch Perdersen site.
  2. We refrain from doing anything until the Manuscript department has decided on whether to adopt some other description standard (Ivan Boserup has proposed the use of TEI P5 Manuscript description which is used by Arna Magnusson Collection of Copenhagen University and Lund university library in St Laurentius Digital Manuscript Library). We do not see any reason to why such more specialist formats not could be used together with (or perhaps even transformed into) METS.

A text critical edition of H.C. Andersens dairies has been digitized. The content comprises 12 volumes of text and digital facsimiles of this modern printed text.

Stake holders

  • DUP Developers - development and technical owners of the application
  • Manuscript Department

Current technology

  • The service has many similarities with ADL and is in good condition. The source text is in text encoded initiative XML.
  • The data model is simple in comparison with ADL, comprising only 6 tables, and 4 indexes.
  • Possible to search full text, much like ADL.

Recommendations

For the time being, we recommend that the transfer to the CMS environment is deferred to a later stage, in conjunction with a redesign of the ADL service.

Feeding Primo and the National Danish Dataset

It is essential for this project to be a success, that data is searchable through Primo, and become a part of the National Danish Dataset. The feeding can be done in three ways:
  • Through OAI-PMH
  • Through xml webservices
  • using copy (ftp, scp etc)

OAI-PMH

Open Archive Initiatives Protocol for Harvesting Metadata, is a protocol that allows any remote client to retrieve our metadata, and store them in a local repository, using them for an unspecified purpose. Through OAI we do not give any search interface or any other logic to the client, it is up to the client to interpret the data.

Both Primo and the National Danish Dataset are OAI enabled.

XML Webservices

XML webeservices is another way of feeding data to remote clients. The webservice lets clients interact (search, browse, view metainfo, etc) with the data, not copy it to a local repository.

The National Danish Dataset might accept a XML webservice, Primo doesn't

Copy

The last way of feeding data, is by simply copy it to a remote client.

Primo accepts this, and any user of the National Danish Dataset might also. However every time the local data is updated, manually copying is required

Recommendations

We recommend that an OAI server for all the data. Since it seems to be a good way of exposing the data to a number of clients. Any other solution will require implementation of more than one service
