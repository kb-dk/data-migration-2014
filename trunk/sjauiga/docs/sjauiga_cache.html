METS File cache

Currently the METS files are stored on img.kb.dk in a file system

      webext.kb.dk:/data1/kb/httpd/img.kb.dk.80/docs/mets/lum    
      webext.kb.dk:/data1/kb/httpd/img.kb.dk.80/docs/mets/musik   
      webext.kb.dk:/data1/kb/httpd/img.kb.dk.80/docs/mets/manus  
      webext.kb.dk:/data1/kb/httpd/img.kb.dk.80/docs/mets/mus
      webext.kb.dk:/data1/kb/httpd/img.kb.dk.80/docs/mets/musman

Each document is in a directory of its own, named after the documents ID in the original application. The cache is built using the get-mets-files.pl client which is installed on img.kb.dk.

Drawbacks with the cache

We are storing these in a web server, and the retrieval of a document is swift and easy. It is, however, computaionally expensive to parse XML, and we consider solutions that permit us to store them in parsed condition, as static DOM objects.

We made some quick and dirty experiments with a Java object cache oscache, but to us it seemed to be about as expensive to read a DOM from the cache as parse a document retrieved via HTTP from a static repository.

Alternatives are to use XML native database (eg xmldb) or put them back into Oracle and XML Text
